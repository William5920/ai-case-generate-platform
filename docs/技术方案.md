# 智能测试用例平台 — 技术方案

---

## 一、项目概述

### 1.1 项目名称

智能测试用例平台（AI Case Generate Platform）

### 1.2 项目目标

为银行软件测试人员提供 AI 驱动的测试用例生成平台。通过上传需求文档或输入一句话需求，自动完成需求文档标准化、需求拆分、测试点提取和测试用例生成，并以脑图形式直观展示需求-测试点-测试用例的层级关系。

### 1.3 核心价值

| 价值维度 | 说明 |
|----------|------|
| **提高效率** | AI 自动生成测试点和测试用例，大幅减少手工编写时间 |
| **质量保障** | 基于标准化需求文档生成，确保测试覆盖全面 |
| **可视化管理** | 脑图形式直观展示需求→测试点→测试用例的层级关系 |
| **可追溯性** | 版本管理保留完整的需求演进过程 |

### 1.4 功能模块

平台包含六大核心模块：

| 模块 | 功能简介 |
|------|----------|
| **用户认证模块** | 用户注册、登录、JWT Token 管理、数据隔离 |
| **需求标准化模块** | 需求录入（文本/文档）、AI需求探索、文档标准化、需求拆分、历史记录 |
| **测试设计模块** | 脑图展示（4级结构）、测试点/用例管理、AI生成、异步任务管理、Excel导出 |
| **知识库模块** | 文档上传/管理、自动切片、向量化入库（Milvus）、知识召回设置与测试 |
| **AI Agent层** | LangGraph 驱动的多步推理Agent，集成知识库RAG |
| **外部AI服务** | OpenAI API对接，支持GPT-4o/GPT-4o-mini等模型 |

### 1.5 业务流程

```
用户注册/登录 → 需求录入（文本/文档）→ 选择模板 → AI需求探索 → 文档标准化
    → AI文档调整 → 需求拆分 → 导入测试设计 → 知识库辅助生成测试点/用例
    → 脑图可视化 → 手动/AI调整 → 导出Excel
```

## 二、总体设计

### 2.1 架构概览

本项目采用前后端分离架构：前端为 Vue 2 + Vuex + Vue Router 的 SPA 应用，后端为 Python 3.11 + FastAPI 的 RESTful API 服务。AI 能力基于 LangGraph Agent + OpenAI API + Milvus 向量数据库实现。

```

│                         前端 (Vue 2)                          │
│  ┌──────────┬──────────────┬──────────────┬──────────────┐   │

│  └──────────┴──────────────┴──────────────┴──────────────┘   │
│                     HTTP REST API (Axios)                     │
├──────────────────────────────────────────────────────────────┤

```

## 三、详细设计

### 3.1 用户认证模块

#### 3.1.1 数据库设计

**users 表：**

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | VARCHAR(36) | PK | UUID 主键 |
| username | VARCHAR(20) | UNIQUE, NOT NULL | 用户名，3-20位 |
| hashed_password | VARCHAR(255) | NOT NULL | bcrypt 加密密码 |
| is_active | BOOLEAN | DEFAULT TRUE | 是否激活 |
| created_at | TIMESTAMP | NOT NULL | 创建时间 |
| updated_at | TIMESTAMP | NOT NULL | 更新时间 |

#### 3.1.2 JWT Token 设计

| 属性 | Access Token | Refresh Token |
|------|-------------|---------------|
| 有效期 | 2 小时 | 7 天 |
| 载荷 | sub(user_id), exp | sub(user_id), exp, type=refresh |
| 用途 | API 请求认证 | 刷新 Access Token |
| 存储 | 前端 localStorage | 前端 localStorage |

#### 3.1.3 API 接口

| 接口 | 方法 | 路径 | 认证 | 说明 |
|------|------|------|------|------|
| 用户注册 | POST | /api/auth/register | 否 | 注册新用户，自动登录返回Token |
| 用户登录 | POST | /api/auth/login | 否 | 验证身份，返回Access/Refresh Token |
| 用户登出 | POST | /api/auth/logout | 是 | 前端清除Token |
| 获取用户信息 | GET | /api/auth/user | 是 | 获取当前登录用户信息 |
| 刷新Token | POST | /api/auth/refresh | 否 | 用RefreshToken换取新AccessToken |

#### 3.1.4 统一响应格式

```json
{
  "code": 200,
  "message": "操作成功",
  "data": { ... }
}
```

### 3.2 需求标准化模块

#### 3.2.1 核心流程

```
需求录入（文本/文档）→ 选择模板 → AI需求探索 → 文档标准化 → AI调整 → 需求拆分
```

- **需求录入**：支持文本直接输入或上传 docx/xlsx/pdf/md 文档
- **模板选择**：支持瀑布式 SRS 模板和敏捷式 User Story 模板
- **AI需求探索**：AI 基于模板维度逐个提问，达到理解度≥80%后可触发文档生成
- **文档标准化**：AI 生成标准化需求规格说明书，在预览区展示
- **AI文档调整**：多轮对话式文档调整，支持采纳/拒绝AI建议
- **需求拆分**：将标准化文档拆分为单个可执行需求项

#### 3.2.2 数据库设计（核心表）

| 表名 | 用途 |
|------|------|
| requirements | 需求主表（标题、输入模式、模板、标准化内容、状态等） |
| uploaded_files | 上传文件表（文件名、路径、大小、用途） |
| explore_messages | AI需求探索对话记录 |
| adjust_messages | AI文档调整对话记录 |
| doc_versions | 文档版本管理（每次采纳建议生成新版本） |
| split_requirements | 拆分后的需求项 |

#### 3.2.3 API 接口总览

| 模块 | 接口数 | 核心能力 |
|------|--------|----------|
| 需求管理 | 5个 | 创建、列表、详情、更新、删除 |
| 模板管理 | 3个 | 列表、详情、AI推荐 |
| AI需求探索 | 4个 | 启动、对话、历史、状态查询 |
| 文档标准化 | 2个 | 执行标准化、获取结果 |
| AI文档调整 | 4个 | 对话、历史、采纳、拒绝 |
| 版本管理 | 4个 | 列表、详情、恢复、差异对比 |
| 需求拆分 | 6个 | 执行拆分、列表、增删改、确认入测试设计 |
| 质量评分 | 1个 | 完整性/清晰度/一致性三维度评分 |

### 3.3 测试设计模块

#### 3.3.1 脑图结构（4级节点）

| 层级 | 节点类型 | 说明 | 视觉样式 |
|------|----------|------|----------|
| 根节点 | 入口需求 | 需求文本简称 | 蓝色渐变背景圆角卡片 |
| 二级节点 | 需求 | 拆分后的单个需求 | 浅蓝背景，蓝色边框 |
| 三级节点 | 测试点 | 提取的测试点 | 浅绿背景，绿色边框；标记保留时显示📌 |
| 四级节点 | 测试用例 | 生成的测试用例 | 白色背景，灰色边框；左侧显示正例/反例标签 |

#### 3.3.2 核心功能

| 功能 | 说明 |
|------|------|
| 快速生成 | 一键触发所有需求的测试点/用例生成任务，串行执行、深度优先 |
| 知识库开关 | 开启后生成时检索知识库内容作为参考 |
| 进度跟踪 | 实时进度条 + 进度描述文本，每2秒轮询任务状态 |
| 取消生成 | 支持在生成中取消任务 |
| 节点操作 | 右键菜单（添加/编辑/删除）、双击编辑、标记保留 |
| AI调整 | 支持对特定节点启动多轮对话调整，标记保留的节点不被替换 |
| 批量操作 | 批量删除、批量标记 |
| 导出Excel | 导出测试用例，含单元格合并，文件命名 `{需求标题}_测试用例.xlsx` |

#### 3.3.4 API 接口总览

| AI调整 | 3个 | 发起对话、发送消息、获取历史 |

### 3.4 知识库模块

#### 3.4.1 核心流程

```
上传文档 → 自动切片 → 向量化入库 → 知识就绪 → 测试设计模块引用
```

#### 3.4.2 功能设计

| 功能 | 说明 |
|------|------|
| 文档上传 | 支持 docx/xlsx/pdf/txt/md，单文件≤50MB，总数≤100个，总空间≤2GB |
| 自动处理 | 上传后自动切片+向量化，状态流转：上传中→切片中→向量化中→已就绪/处理失败 |
| 状态轮询 | 非终态文档每3秒轮询一次状态 |
| 失败重试 | 最多重试3次 |
| 统计面板 | 文档总数、已就绪数、处理中数、存储空间使用率 |
| 召回设置 | 可调整 top_k、相似度阈值、切片大小/重叠、召回策略（混合/纯向量） |
| 召回测试 | 输入查询文本，查看检索结果验证召回质量 |

#### 3.4.3 API 接口总览

| 模块 | 接口数 | 核心能力 |
|------|--------|----------|
| 文档管理 | 8个 | 列表、详情、上传、删除、重试、切片查看、内容查看、存储信息 |
| 处理状态 | 2个 | 单个查询、批量查询 |
| 召回设置 | 3个 | 获取、更新、触发重新处理 |
| 召回测试 | 2个 | 执行测试、获取历史 |
| 统计信息 | 2个 | 文档统计、处理状态统计 |

## 四、AI 技术应用方案

### 4.1 整体架构：LangGraph Agent 编排

摒弃传统单轮 LLM 调用方式，采用 LangGraph StateGraph 构建多步推理智能体，实现分析→召回→生成→校验→修正的完整闭环。

```
┌─────────────────────────────────────────────────────────────┐
│                    TestDesignOrchestrator                     │
│                   (编排层，替代原 _run_generation)             │
├────────────────────────┬────────────────────────────────────┤
│                        │                                    │
│  ┌─────────────────────▼──────────────────────┐             │
│  │         TestPointAgent (LangGraph)          │             │
│  │  analyze_requirement → recall_knowledge     │             │
│  │       → generate → self_review              │             │
│  │       → [refine] → output                   │             │
│  └─────────────────────┬──────────────────────┘             │
│                        │ 测试点列表                           │
│  ┌─────────────────────▼──────────────────────┐             │
│  │         TestCaseAgent (LangGraph)           │             │
│  │  analyze_test_point → recall_knowledge      │             │
│  │       → generate → quality_check            │             │
│  │       → [self_correct] → output             │             │
│  └─────────────────────────────────────────────┘             │
├──────────────────────────────────────────────────────────────┤
│                    共享基础设施层                              │
│  ┌────────────┐  ┌────────────┐  ┌─────────────────────┐   │
│  │ LLM Client │  │ RAG Service│  │ KnowledgeBaseService │   │
│  │ (OpenAI)   │  │ (召回+重排) │  │ (Milvus)            │   │
│  └────────────┘  └────────────┘  └─────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 Agent 详细设计

#### 4.2.1 TestPointAgent

| 步骤 | 节点 | 功能 | 模型 |
|------|------|------|------|
| 1 | analyze_requirement | 分析需求文本，提取关键测试维度 | gpt-4o-mini |
| 2 | recall_knowledge | 调用 RAG 检索相关知识（useKnowledgeBase=true时） | - |
| 3 | generate | 基于分析结果+知识库内容生成测试点 | gpt-4o |
| 4 | self_review | 评审生成的测试点（覆盖度、合理性） | gpt-4o-mini |
| 5 | refine | 根据评审反馈修正（最多2轮） | gpt-4o |

#### 4.2.2 TestCaseAgent

| 步骤 | 节点 | 功能 | 模型 |
|------|------|------|------|
| 1 | analyze_test_point | 分析测试点，推导测试场景 | gpt-4o-mini |
| 2 | recall_knowledge | 调用 RAG 检索相关领域知识 | - |
| 3 | generate | 生成正例/反例测试用例（含前置条件、步骤） | gpt-4o |
| 4 | quality_check | 校验用例完整性、步骤合理性、正反例均衡 | gpt-4o-mini |
| 5 | self_correct | 根据校验结果修正（最多2轮） | gpt-4o |

### 4.3 知识库 RAG 集成

#### 4.3.1 文档处理流程

```
上传文档 → 解析提取文本 → 文本切片(chunk_size=500, overlap=50) → Embedding向量化 → 存入Milvus
```

#### 4.3.2 召回策略

| 策略 | 实现方式 | 适用场景 |
|------|----------|----------|
| vector | Milvus 向量相似度搜索 | 语义相近但用词不同的内容 |
| hybrid | 向量搜索 + BM25 关键词匹配，加权融合 | 包含专业术语、编号规则的内容 |
| rerank | 向量搜索后 LLM 对 top_k 结果重排序 | 需要高精度召回的场景 |

#### 4.3.3 LLM Client 封装

- 使用 OpenAI `response_format: {"type": "json_object"}` 强制 JSON 输出
- system prompt 中嵌入 JSON Schema 描述确保格式稳定
- 内置指数退避重试（最多3次），处理 rate limit
- 成本优化：gpt-4o-mini 用于分析/校验步骤，gpt-4o 用于核心生成步骤

### 4.4 AI 需求探索与标准化

#### 4.4.1 需求探索

- AI 基于选中模板的维度（如 SRS 的6章+附录）逐个提问
- 实时计算需求理解度 = 已覆盖维度 / 总维度数
- 达到 80% 后 AI 主动提示可开始生成文档
- 支持快捷回复按钮和自由文本输入

#### 4.4.2 文档标准化

- 结合原始需求文本和探索对话中确认的信息生成标准化文档
- AI 调整支持采纳/拒绝建议，采纳后自动生成新版本
- 质量评分：完整性、清晰度、一致性三维度，0-100分

### 4.5 异步任务管理

- 快速生成采用串行执行 + 深度优先策略
- 进度计算：`progress = completed_items / total_items * 100`
- 前端每 2 秒轮询任务状态接口
- 支持任务取消（调用取消接口，停止轮询，重置状态）
- 任务完成后自动更新需求状态为"已完成"

## 六、经验总结

### 6.1 LangGraph Agent 设计心得

1. **多步推理优于单轮调用**：传统单 prompt 方式生成的测试用例质量不稳定，通过 LangGraph StateGraph 将流程拆分为分析→召回→生成→校验→修正，每一步有明确的输入输出，质量显著提升。

2. **模型分层使用降本增效**：分析/校验步骤使用 gpt-4o-mini（成本约为 gpt-4o 的 1/10），仅核心生成步骤使用 gpt-4o，在保证质量的同时大幅降低 API 调用成本。

3. **JSON Schema 约束是必要手段**：尽管使用了 `response_format: json_object`，仍需在 system prompt 中嵌入详细 JSON Schema 描述才能保证输出格式稳定。同时保留正则提取 JSON 作为兜底方案。

4. **自我修正不宜过多轮次**：Agent 的 refine/self_correct 步骤设定为最多 2 轮，超过后强行终止。实践中发现超过 2 轮的修正边际收益递减，且可能引入新的幻觉。

### 6.2 知识库 RAG 集成经验

1. **切片策略需要业务适配**：通用 500 字符 + 50 字符重叠的切片策略对银行测试文档效果好，但需要根据文档类型调整。对于表格密集的文档，需要按表格行切分。

2. **混合召回优于纯向量召回**：向量搜索对语义理解好但对精确术语匹配差（如 "SWIFT MT103"），结合 BM25 关键词匹配的混合召回能覆盖更多场景。

3. **LLM 重排序是性价比高的优化**：先召回 top_k=5→10 条，再用 gpt-4o-mini 重排序选出 top_k=3 条，成本可控且效果显著。

### 6.3 异步任务管理经验

1. **串行执行 + 深度优先** 是合理的策略：单个需求→多个测试点→多个测试用例，逐深度完成再进入下一需求。避免并发调用 LLM 导致的 rate limit 问题，且进度可精确追踪。

2. **轮询机制设计**：2 秒轮询间隔是体验与服务器压力的平衡点。任务状态变更后立即更新进度百分比和描述文本，让用户感知到"正在工作"。

3. **取消机制**：取消不是即时生效的——需要等当前正在执行的 LLM 调用完成后才能安全停止。取消后数据库中的中间数据可选择保留或清理。

### 6.4 前端脑图集成经验

1. **simple-mind-map** 组件选型：支持自定义节点渲染、右键菜单、展开/折叠、缩放拖拽，满足 4 级节点结构需求。但自定义节点样式（如图钉标记、来源标签）需要深入组件内部修改。

2. **数据与视图分离**：脑图数据由后端以树形 JSON 返回，前端仅做渲染。任何编辑操作都通过 API 持久化后再刷新视图，保证数据一致性。

3. **Excel 导出复杂度**：含步骤的测试用例场景下，需要单元格合并（caseName/caseProperty/preCondition 跨行合并），openpyxl 的 merge_cells 配合行号计算实现。

### 6.5 中间件运维经验

1. **Docker 本地部署降低环境成本**：MySQL 和 Milvus 均通过 Docker 一键部署，开发/测试环境一致，减少"在我机器上能跑"问题。

2. **Milvus Standalone 模式足够**：对于单用户场景，Standalone 模式（单节点）足够满足向量检索需求。分布式模式（Cluster）仅在数据量达到千万级以上才需要。

3. **国内网络环境**：Docker 镜像拉取建议配置国内镜像加速器（中科大/网易/阿里云），Milvus 镜像较大（约 2GB），首次部署需预留足够时间。

4. **taisacle 异地联调**：当后端和数据库部署在本地、前端部署在云服务器或需要异地进行联调时，taisacle 提供了便捷的内网穿透方案。使用时需注意配置 CORS 白名单，且 Milvus SDK 连接需要直接指定 Standalone 地址而非通过服务发现。

### 6.6 项目管理经验

1. **先文档后代码**：完整的需求文档（PRD）、接口文档（API Spec）、后端规划文档（Backend Plan）和前端规划文档（Frontend Plan）为开发提供了清晰的执行蓝图，减少了返工。

2. **模块化拆分**：将平台按功能拆分为认证、需求标准化、测试设计、知识库四个独立模块，各模块可并行开发和测试，最后通过统一的路由和状态管理集成。

3. **设计规范的统一**：制定完整的设计规范（色彩系统、排版系统、间距系统、组件规范），确保多页面/多组件开发时风格一致。
